01 Walk me through your project

Q
"Can you walk me through what you built and how you approached it?"

Start with the problem, then the decision, then the result.

Say this
"The requirement was to build a system that reads data files and a logic file, then answers any question about college students correctly using a free model — specifically something like NotebookLM.

My approach was to mirror exactly how NotebookLM works internally. NotebookLM is a Google product — it loads your full documents into a Gemini context window and lets the model reason over everything at once. I applied the same architecture to structured data: load all 12 CSV datasets into SQLite, serialize every table as plain text, pre-compute critical aggregates in SQL so the LLM reads verified facts, and send the complete picture to Gemini 1.5 Flash — the same free model that powers NotebookLM.

The result is a Streamlit web app where you type any question and get a grounded, numbered, reasoned answer — no hallucination, because every number the model cites comes directly from the data."

02 Requirements — how each one is satisfied

Input = a data file
12 CSV files in data/ folder
load_data.py reads every CSV on startup, auto-detects column types (INTEGER / REAL / TEXT), and loads them into student_intelligence.db — a SQLite file. Any new CSV can be uploaded live from the app UI without restarting.
Input = a logic file
The system prompt in llm.py
The SYSTEM_PROMPT constant is a separate logic file. It contains 5 analytical rules: scan all data first, find signal not noise, cite exact numbers, rank everyone (not just the winner), explain the WHY. It governs how the LLM reasons — completely separate from the data files.
Everything about college students — correct answers
12 tables + pre-computed summaries
Covers grades, attendance, study hours, mid/end sem marks, assignments, projects, placement readiness. The _build_computed_summaries() function pre-calculates attendance %, averages, and rankings in Python via SQL — these verified facts are appended to the context so the LLM reads answers, not raw rows to count.
Use a free model
Gemini 1.5 Flash — free tier
Google AI Studio provides Gemini 1.5 Flash completely free — no credit card, no billing. 1 million token context window. If it rate-limits, the system automatically falls back to Groq's Llama-3.3-70B, also free. Both keys go in a .env file.
NotebookLM-inspired
Same model. Same architecture.
NotebookLM runs on Gemini — I used the same model family. NotebookLM loads full documents into one context window — I load full tables the same way. The app UI mirrors NotebookLM's Sources sidebar. The approach is architecturally identical, adapted for structured CSV data instead of PDFs.
Add study hours + mid/end sem marks
3 new datasets added
study_hours.csv (35 sessions across all students), mid_sem_marks.csv (marks out of 50), end_sem_marks.csv (marks out of 100). The engine cross-joins these to compute mid→end improvement per student automatically.

03 The NotebookLM question — nail this one

Q
"I keep saying NotebookLM — how does your system relate to it?"

This is the most important question. The interviewer mentioned this multiple times — she wants to see you understand why NotebookLM is the reference, not just that you used it as a buzzword.

Say exactly this
"NotebookLM is built on Gemini — that's Google's model. When you upload documents to NotebookLM, it doesn't chop them into pieces and search for relevant fragments. It loads your entire document into Gemini's context window and lets the model read everything at once before answering.

I applied that exact same idea to structured data. Instead of PDF documents, my source is 12 CSV tables. My data_engine.py serializes every table row-by-row into plain text — like converting a spreadsheet into a readable document — and passes it all to Gemini in one call.

I even use the same model — Gemini 1.5 Flash, free via Google AI Studio. The UI mirrors NotebookLM's layout too: a Sources panel on the left showing all loaded datasets, and a main area for questions and answers.

So in one line: NotebookLM for structured college data, using the same model, same architecture, zero cost."
Why this impresses
You're showing you understand the principle behind NotebookLM (full-context reasoning), not just the name. Most people would say "I got inspired by the interface." You're saying "I understood the architecture and replicated it for a different data type."

04 Technical questions — be ready for these

Q
"How do you make sure the answers are correct? What prevents wrong output?"
Say this
"Two mechanisms. First, the system prompt has an absolute rule: every number cited must exist verbatim in the data. If the information isn't there, the model must say so. No estimation, no inference from partial data.

Second — and this is the important one — I pre-compute critical aggregates in Python using SQL before the LLM ever sees the data. Things like attendance percentage per student, average scores, mid-to-end-sem improvement. A large language model can miscount rows or miss a SQL join if it's trying to do that arithmetic on its own. By running it in Python first and feeding the model verified numbers, I remove that error source entirely. The LLM reads facts, not raw data to calculate from."
Q
"Why didn't you use RAG — vector embeddings and semantic search?"
Say this
"RAG makes sense when the dataset is too large to fit in a context window — like a 500-page PDF library. But my dataset is a few hundred rows across 12 tables, which fits comfortably in Gemini 1.5 Flash's 1-million-token window.

RAG would actually hurt here. If someone asks 'who is the best student overall,' a retrieval system might return only the top-scores table and miss the attendance data, or miss the study hours. By loading everything at once, the model sees the full picture for every question — which is exactly how NotebookLM works too. Full context beats partial retrieval when you have the context budget for it."
Q
"What is the 'logic file' you built?"
Say this
"The requirement said 'logic is given in another file' — separate from the data files. In my system, this is the SYSTEM_PROMPT constant in llm.py. It contains five analytical reasoning rules that govern how the LLM approaches every question:

1. Scan the full picture first — never jump to a conclusion from one column
2. Find signal not noise — a 4.5 vs 4.7 feedback gap is noise; a 74 vs 89 score gap is signal
3. Tell the story with numbers — vague language like 'performed well' without a number is not allowed
4. Compare everyone — always show the full ranking, not just the winner
5. Explain the WHY — connect study hours to grades, attendance patterns to performance

That is the logic the model runs on, completely separate from the data."
Q
"What datasets did you use? Why did you add more?"

The original 4 given were: student_results, attendance, department_location, teaching_efficiency.

Say this
"The given datasets covered grades and attendance, but to answer richer questions — like 'which student improved the most?' or 'is there a link between study hours and grades?' — I needed more depth. So I added 8 more tables:

The three you specifically mentioned: study_hours (daily study sessions per subject), mid_sem_marks (marks out of 50), and end_sem_marks (marks out of 100). The engine automatically computes the improvement from mid to end sem per student and per course.

I also added student_profile (names, CGPA, mentor), assignment_scores, project_performance, placement_readiness, and teacher_profile — making the system able to answer career-level, behavioral, and cross-dimensional questions, not just 'what score did this student get.'"
Q
"Walk me through one question end-to-end — how does the system process it?"
1
User types: "Which student improved the most from mid-sem to end-sem?"
Streamlit captures the question in app.py and calls narrate(question, ctx)
2
data_engine loads all context
load_all_context() serializes all 12 tables as readable text. Then _build_computed_summaries() runs a SQL JOIN between mid_sem_marks and end_sem_marks, computes average improvement per student, and appends the ranked result.
3
LLM call with full context
llm.py sends system prompt + all data + the question to Gemini 1.5 Flash in a single HTTP call. The model reads the pre-computed improvement rankings and reasons over them.
4
Answer rendered with citations
The answer appears in a styled card. Citation chips at the bottom show which tables were referenced — e.g. mid_sem_marks, end_sem_marks, student_profile.
